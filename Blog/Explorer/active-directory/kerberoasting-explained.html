Kerberoasting Explained

Why any authenticated domain user can request a crackable hash for any service account — and how to actually pull it off.

kerberosactive directorycredential access

The core idea

Any account with a Service Principal Name (SPN) set — typically service accounts running SQL Server, IIS app pools, scheduled tasks, etc. — can be asked for a Kerberos service ticket by any authenticated domain user, no matter how low-privileged. That ticket is encrypted with the service account's own password hash. Take the ticket offline and crack it, and you've recovered the service account's password without ever touching the DC's authentication logs beyond a normal ticket request.

Why it works

Kerberos issues a Ticket Granting Service (TGS) ticket for the requested SPN, encrypted using the target service account's NTLM hash — not the requester's. If that hash derives from a weak password, it's crackable offline, at whatever speed the attacker's hardware allows.

Execution

# enumerate SPNs and request tickets in one pass
impacket-GetUserSPNs htb.local/lowpriv:'password' -dc-ip 10.10.10.5 -request

# crack offline
hashcat -m 13100 spns.hash rockyou.txt

Why service accounts are the weak point

  • Passwords are often set once at provisioning and never rotated.
  • They're rarely subject to the same complexity/length policy enforcement as interactive user accounts in practice.
  • Privilege is often broad — service accounts backing SQL Server or IIS frequently carry local admin rights on multiple hosts.

Defensive notes

Long, randomly generated service account passwords (25+ characters) make offline cracking impractical. Group Managed Service Accounts (gMSAs) remove the problem entirely — Windows rotates their password automatically and it's never human-memorable, so there's nothing crackable to recover.
