IDOR Hunting Methodology

A repeatable process for finding Insecure Direct Object Reference bugs instead of stumbling into them by accident.

idorauthzmethodology

1. Map every object reference

Walk the app as two separate low-privileged users and log every place an ID, slug, UUID, or filename shows up — in the URL, in JSON bodies, in headers, even in filenames for uploads/downloads. Anything that identifies "which record" is a candidate.

2. Swap and observe

For each candidate, replace User A's ID with User B's while authenticated as A. Check for three outcomes, not just one:

  • Full disclosure — the other user's data comes back unchanged.
  • Partial disclosure — a 200 with a trimmed-down or different-shaped response (still a finding, easy to miss with only a status-code diff).
  • Blind write — a 200/204 on a POST/PUT/DELETE with no reflected data at all; confirm impact by re-fetching as the victim.

3. Don't stop at numeric IDs

UUIDs and hashes get treated as "unguessable" and skipped — but if they leak anywhere else in the app (another endpoint, an email, a PDF export, a support ticket), the "randomness" stops mattering. Also check GraphQL node IDs and batched/array parameters, which frequently skip the authz check applied to their singular counterparts.

4. Check every HTTP verb per endpoint

Authz checks are often bolted onto GET and forgotten on PUT/PATCH/DELETE for the same resource. Re-test the full verb set for any endpoint that returns object data.

Quick Burp workflow

# Autorize / Auth Analyzer extension:
1. Log requests as low-priv user A
2. Replay each with cookie/token swapped to user B (and vice versa)
3. Diff response length + status code, then manually confirm anything "unauthorized"
