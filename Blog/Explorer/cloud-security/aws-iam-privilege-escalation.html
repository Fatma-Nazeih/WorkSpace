AWS IAM Privilege Escalation Paths

A working list of IAM permission combinations that let a low-privileged identity escalate to admin — useful once initial access to any AWS credential is gained.

awsiamprivesc

Why this matters

Most cloud engagements don't start with root credentials — they start with one leaked key or an over-permissioned Lambda role. The interesting part is what that identity can reach from there. IAM privesc is rarely one obvious hole; it's usually a chain of permissions that look harmless individually.

High-value permission combos

Permission(s)Path to escalation
iam:CreatePolicyVersionAttach a new admin policy version to a role/user you control
iam:AttachUserPolicyAttach AdministratorAccess directly to yourself
iam:PassRole + lambda:CreateFunctionCreate a Lambda with an over-privileged execution role, invoke it
iam:PassRole + ec2:RunInstancesLaunch an EC2 instance with an admin instance profile, pull creds from metadata
iam:CreateAccessKeyMint new keys for a higher-privileged user
iam:UpdateAssumeRolePolicyModify a role's trust policy to allow your principal to assume it

Enumeration first

Before chasing exploitation, map what the current identity can actually do:

aws sts get-caller-identity
aws iam list-attached-user-policies --user-name <user>
aws iam simulate-principal-policy --policy-source-arn <arn> --action-names iam:PassRole

Tools like Pacu and PMapper automate a lot of this graph-building — worth running early rather than eyeballing individual policies.

Notes

  • CloudTrail logs every one of these actions — treat privesc testing on live environments as loud by default.
  • Wildcard resources ("Resource": "*") on iam:* actions are the single most common root cause when reviewing policies.
